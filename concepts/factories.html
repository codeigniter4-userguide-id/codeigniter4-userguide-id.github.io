Pendahuluan

Apa itu Factories?

Seperti Service, Factories adalah ekstensi dari autoloading yang membantu menjaga kode Anda tetap ringkas namun optimal, tanpa harus meneruskan instance objek antar class.

Factories mirip dengan $this->load di CodeIgniter 3 dalam poin-poin berikut:

  • Memuat class

  • Berbagi instance class yang dimuat

Pada dasarnya, Factories menyediakan cara umum untuk membuat instance class dan mengaksesnya dari mana saja. Ini adalah cara yang bagus untuk menggunakan kembali status objek dan mengurangi beban memori dari menjaga beberapa instance dimuat di seluruh aplikasi Anda.

Class apa pun dapat dimuat oleh Factories, tetapi contoh terbaik adalah class yang digunakan untuk bekerja atau mentransmisikan data umum. Framework itu sendiri menggunakan Factories secara internal, mis., untuk memastikan konfigurasi yang benar dimuat saat menggunakan class Config.

Perbedaan dari Services

Factories memerlukan nama class konkret untuk diinstansiasi dan tidak memiliki kode untuk membuat instance.

Jadi, Factories tidak baik untuk membuat instance kompleks yang membutuhkan banyak dependensi, dan Anda tidak dapat mengubah class instance yang akan dikembalikan.

Di sisi lain, Services memiliki kode untuk membuat instance, sehingga dapat membuat instance kompleks yang membutuhkan service atau instance class lain. Ketika Anda mendapatkan service, Services memerlukan nama service, bukan nama class, sehingga instance yang dikembalikan dapat diubah tanpa mengubah kode klien.

Memuat Class

Memuat Sebuah Class

Lihatlah Models sebagai contoh. Anda dapat mengakses Factory khusus untuk Model dengan menggunakan metode statis ajaib dari class Factories, Factories::models().

Nama metode statis disebut component.

Meneruskan Nama Class tanpa Namespace

Jika Anda meneruskan nama class tanpa namespace, Factories pertama-tama mencari di namespace App untuk jalur yang sesuai dengan nama metode statis ajaib. Factories::models() mencari di direktori app/Models.

Meneruskan Nama Class Pendek

Dalam kode berikut, jika Anda memiliki App\Models\UserModel, instance akan dikembalikan:

<?php

use CodeIgniter\Config\Factories;

$users = Factories::models('UserModel');

Jika Anda tidak memiliki App\Models\UserModel, ia mencari Models\UserModel di semua namespace.

Lain kali Anda meminta class yang sama di mana pun dalam kode Anda, Factories akan memastikan Anda mendapatkan kembali instance seperti sebelumnya:

<?php

use CodeIgniter\Config\Factories;

class SomeOtherClass
{
    public function someFunction()
    {
        $users = Factories::models('UserModel');

        // ...
    }
}
Meneruskan Nama Class Pendek dengan Sub-direktori

Jika Anda ingin memuat class di sub-direktori, Anda menggunakan / sebagai pemisah. Kode berikut memuat app/Libraries/Sub/SubLib.php jika ada:

use CodeIgniter\Config\Factories;

$lib = Factories::libraries('Sub/SubLib');

Meneruskan Nama Class yang Sepenuhnya Berkualifikasi

Anda juga dapat meminta nama class yang sepenuhnya berkualifikasi:

use CodeIgniter\Config\Factories;

$users = Factories::models('Blog\Models\UserModel');
// Or
$users = Factories::models(\Blog\Models\UserModel::class);

Ini mengembalikan instance Blog\Models\UserModel jika ada.

Catatan

Sebelum v4.4.0, ketika Anda meminta nama class yang sepenuhnya berkualifikasi, jika Anda hanya memiliki Blog\Models\UserModel, instance akan dikembalikan. Tetapi jika Anda memiliki keduanya App\Models\UserModel dan Blog\Models\UserModel, instance dari App\Models\UserModel akan dikembalikan.

Jika Anda ingin mendapatkan Blog\Models\UserModel, Anda perlu menonaktifkan opsi preferApp:

use CodeIgniter\Config\Factories;

$users = Factories::models('Blog\Models\UserModel', ['preferApp' => false]);

Fungsi Kemudahan

Dua fungsi pintasan untuk Factories telah disediakan. Fungsi-fungsi ini selalu tersedia.

config()

Yang pertama adalah config() yang mengembalikan instance baru dari class Config. Satu-satunya parameter yang diperlukan adalah nama class:

<?php

$appConfig = config('App');

// The code above is the same as the code below.
$appConfig = \CodeIgniter\Config\Factories::config('App');

model()

Fungsi kedua, model() mengembalikan instance baru dari class Model. Satu-satunya parameter yang diperlukan adalah nama class:

<?php

$user = model('UserModel');

// The code above is the same as the code below.
$user = \CodeIgniter\Config\Factories::models('UserModel');

Mendefinisikan Nama Class yang Akan Dimuat

Baru pada versi 4.4.0.

Anda dapat mendefinisikan nama class yang akan dimuat sebelum memuat class dengan metode Factories::define():

use CodeIgniter\Config\Factories;

Factories::define('models', 'Myth\Auth\Models\UserModel', 'App\Models\UserModel');

Parameter pertama adalah komponen. Parameter kedua adalah alias class (parameter pertama untuk metode statis ajaib Factories), dan parameter ketiga adalah nama class yang sepenuhnya berkualifikasi yang akan dimuat.

Setelah itu, jika Anda memuat Myth\Auth\Models\UserModel dengan Factories, instance App\Models\UserModel akan dikembalikan:

$users = model('Myth\Auth\Models\UserModel');

Parameter Factory

Factories mengambil sebagai parameter kedua array nilai opsi (dijelaskan di bawah). Direktif ini akan menimpa opsi default yang dikonfigurasi untuk setiap komponen.

Parameter lain yang diteruskan pada saat yang sama akan diteruskan ke konstruktor class, memudahkan konfigurasi instance class Anda secara on-the-fly. Misalnya, katakan aplikasi Anda menggunakan database terpisah untuk otentikasi dan Anda ingin memastikan bahwa setiap upaya untuk mengakses catatan pengguna selalu melalui koneksi tersebut:

<?php

use CodeIgniter\Config\Factories;

$conn  = db_connect('auth');
$users = Factories::models('UserModel', [], $conn);

Sekarang setiap kali UserModel dimuat dari Factories, itu akan mengembalikan instance class yang menggunakan koneksi database alternatif.

Opsi Factories

Perilaku default mungkin tidak berfungsi untuk setiap komponen. Misalnya, katakan nama komponen Anda dan jalurnya tidak selaras, atau Anda perlu membatasi instance ke jenis class tertentu. Setiap komponen mengambil serangkaian opsi untuk mengarahkan penemuan dan instansiasi.

Perilaku Factories

Opsi dapat diterapkan dalam salah satu dari tiga cara (terdaftar dalam prioritas naik):

  • Class konfigurasi Config\Factory dengan properti yang cocok dengan nama komponen.

  • Metode statis Factories::setOptions().

  • Meneruskan opsi langsung pada waktu panggilan dengan parameter.

Konfigurasi

Untuk mengatur opsi komponen default, buat file Config baru di app/Config/Factory.php yang menyediakan opsi sebagai properti array yang cocok dengan nama komponen.

Contoh: Filters Factories

Misalnya, jika Anda ingin membuat Filters oleh Factories, nama komponen akan menjadi filters. Dan jika Anda ingin memastikan bahwa setiap filter adalah instance dari class yang mengimplementasikan FilterInterface CodeIgniter, file app/Config/Factory.php Anda mungkin terlihat seperti ini:

<?php

namespace Config;

use CodeIgniter\Config\Factory as BaseFactory;
use CodeIgniter\Filters\FilterInterface;

class Factory extends BaseFactory
{
    public $filters = [
        'instanceOf' => FilterInterface::class,
    ];
}

Sekarang Anda dapat membuat filter dengan kode seperti Factories::filters('SomeFilter'), dan instance yang dikembalikan pasti akan menjadi filter CodeIgniter.

Ini akan mencegah konflik modul pihak ketiga yang kebetulan memiliki jalur Filters yang tidak terkait di namespace-nya.

Contoh: Library Factories

Jika Anda ingin memuat class library Anda di direktori app/Libraries dengan Factories::library('SomeLib'), jalur Libraries berbeda dari jalur default Library.

Dalam kasus ini, file app/Config/Factory.php Anda akan terlihat seperti ini:

<?php

namespace Config;

use CodeIgniter\Config\Factory as BaseFactory;

class Factory extends BaseFactory
{
    public $library = [
        'path' => 'Libraries',
    ];
}

Sekarang Anda dapat memuat library Anda dengan metode Factories::library():

use CodeIgniter\Config\Factories;

$someLib = Factories::library('SomeLib');

Metode setOptions

Class Factories memiliki metode statis untuk memungkinkan konfigurasi opsi runtime: cukup berikan array opsi yang diinginkan menggunakan metode setOptions() dan mereka akan digabungkan dengan nilai default dan disimpan untuk panggilan berikutnya:

<?php

use CodeIgniter\Config\Factories;
use CodeIgniter\Filters\FilterInterface;

Factories::setOptions('filters', [
    'instanceOf' => FilterInterface::class,
    'prefersApp' => false,
]);

Opsi Parameter

Panggilan statis ajaib Factories mengambil sebagai parameter kedua array nilai opsi. Direktif ini akan menimpa opsi yang disimpan yang dikonfigurasi untuk setiap komponen dan dapat digunakan pada waktu panggilan untuk mendapatkan persis apa yang Anda butuhkan. Input harus berupa array dengan nama opsi sebagai kunci untuk setiap nilai penimpaan.

Misalnya, secara default Factories mengasumsikan bahwa Anda ingin menemukan instance bersama dari komponen. Dengan menambahkan parameter kedua ke panggilan statis ajaib, Anda dapat mengontrol apakah panggilan tunggal itu akan mengembalikan instance baru atau bersama:

use CodeIgniter\Config\Factories;

$users = Factories::models('UserModel', ['getShared' => true]);  // Default; will always be the same instance
$other = Factories::models('UserModel', ['getShared' => false]); // Will always create a new instance

Caching Konfigurasi

Baru pada versi 4.4.0.

Penting

Jangan gunakan fitur ini kecuali Anda telah membaca bagian ini dengan cermat dan memahami cara kerja fitur ini. Jika tidak, aplikasi Anda tidak akan berfungsi dengan benar.

Untuk meningkatkan kinerja, Caching Konfigurasi telah diimplementasikan.

Prasyarat

Penting

Menggunakan fitur ini ketika prasyarat tidak terpenuhi akan mencegah CodeIgniter beroperasi dengan benar. Jangan gunakan fitur ini dalam kasus seperti itu.

  • Untuk menggunakan fitur ini, properti semua objek Config yang diinstansiasi di Factories tidak boleh dimodifikasi setelah instansiasi. Dengan kata lain, class Config harus berupa class immutable atau readonly.

  • Secara default, setiap class Config yang di-cache harus mengimplementasikan metode __set_state().

Bagaimana Cara Kerjanya

Penting

Setelah di-cache, nilai konfigurasi tidak pernah berubah sampai cache dihapus, bahkan jika file konfigurasi atau .env diubah.

  • Simpan semua instance Config di Factories ke dalam file cache sebelum shutdown, jika status instance Config di Factories berubah.

  • Pulihkan instance Config yang di-cache sebelum inisialisasi CodeIgniter jika cache tersedia.

Sederhananya, semua instance Config yang dipegang oleh Factories di-cache segera sebelum shutdown, dan instance yang di-cache digunakan secara permanen.

Bagaimana Cara Memperbarui Nilai Konfigurasi

Setelah disimpan, versi yang di-cache tidak pernah kedaluwarsa. Mengubah file Config yang ada (atau mengubah Variabel Lingkungan untuk itu) tidak akan memperbarui cache maupun nilai Config.

Jadi jika Anda ingin memperbarui nilai Config, perbarui file Config atau Variabel Lingkungan untuk itu, dan Anda harus menghapus file cache secara manual.

Anda dapat menggunakan perintah spark cache:clear:

php spark cache:clear

Atau cukup hapus file writable/cache/FactoriesCache_config.

Catatan

Sejak v4.5.0, perintah spark optimize membersihkan cache.

Bagaimana Cara Mengaktifkan Caching Konfigurasi

Baru pada versi 4.5.0.

Atur properti berikut ke true di app/Config/Optimize.php:

public bool $configCacheEnabled = true;

Atau Anda dapat mengaktifkan ini dengan perintah spark optimize.

Catatan

Properti ini tidak dapat ditimpa oleh variabel lingkungan.
