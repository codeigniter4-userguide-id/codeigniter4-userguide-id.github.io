Panduan Keamanan

Kami menganggap keamanan dengan serius. CodeIgniter menggabungkan sejumlah fitur dan teknik untuk menegakkan praktik keamanan yang baik, atau untuk memungkinkan Anda melakukannya dengan mudah.

Kami merujuk pada Open Web Application Security Project (OWASP) dan mengikuti rekomendasi mereka sebisa mungkin.

Berikut ini berasal dari OWASP Top Ten dan OWASP API Security Top 10 mengidentifikasi kerentanan teratas untuk aplikasi web dan API. Untuk masing-masing, kami memberikan deskripsi singkat, rekomendasi OWASP, dan kemudian ketentuan CodeIgniter untuk mengatasi masalah tersebut.

OWASP Top 10 2021

A01:2021 Kontrol Akses Rusak

Kontrol akses memberlakukan kebijakan sehingga pengguna tidak dapat bertindak di luar izin yang dimaksudkan. Kegagalan biasanya mengarah pada pengungkapan informasi yang tidak sah, modifikasi, atau penghancuran semua data atau melakukan fungsi bisnis di luar batas pengguna.

Kerentanan kontrol akses umum meliputi:

  • Pelanggaran prinsip hak istimewa paling rendah atau penolakan secara default, di mana akses seharusnya hanya diberikan untuk kemampuan, peran, atau pengguna tertentu, tetapi tersedia untuk siapa saja.

  • Melewati pemeriksaan kontrol akses dengan memodifikasi URL (perusakan parameter atau penjelajahan paksa), status aplikasi internal, atau halaman HTML, atau dengan menggunakan alat serangan yang memodifikasi permintaan API.

  • Mengizinkan melihat atau mengedit akun orang lain, dengan menyediakan pengidentifikasi uniknya (referensi objek langsung yang tidak aman)

  • Mengakses API dengan kontrol akses yang hilang untuk POST, PUT, dan DELETE.

  • Peningkatan hak istimewa. Bertindak sebagai pengguna tanpa masuk atau bertindak sebagai admin saat masuk sebagai pengguna.

  • Manipulasi metadata, seperti memutar ulang atau merusak token kontrol akses JSON Web Token (JWT), atau cookie atau bidang tersembunyi yang dimanipulasi untuk meningkatkan hak istimewa atau menyalahgunakan pembatalan JWT.

  • Kesalahan konfigurasi CORS memungkinkan akses API dari asal yang tidak sah/tidak tepercaya.

  • Penjelajahan paksa ke halaman yang diautentikasi sebagai pengguna yang tidak diautentikasi atau ke halaman yang memiliki hak istimewa sebagai pengguna standar.

Rekomendasi OWASP

Kontrol akses hanya efektif dalam kode sisi server tepercaya atau API tanpa server, di mana penyerang tidak dapat memodifikasi pemeriksaan kontrol akses atau metadata.

  • Kecuali untuk sumber daya publik, tolak secara default.

  • Terapkan mekanisme kontrol akses sekali dan gunakan kembali di seluruh aplikasi, termasuk meminimalkan penggunaan Cross-Origin Resource Sharing (CORS).

  • Kontrol akses model harus memberlakukan kepemilikan catatan daripada menerima bahwa pengguna dapat membuat, membaca, memperbarui, atau menghapus catatan apa pun.

  • Persyaratan batas bisnis aplikasi yang unik harus diberlakukan oleh model domain.

  • Nonaktifkan daftar direktori server web dan pastikan metadata file (misalnya, .git) dan file cadangan tidak ada di dalam akar web.

  • Catat kegagalan kontrol akses, peringatkan admin jika sesuai (misalnya, kegagalan berulang).

  • Batasi tingkat akses API dan controller untuk meminimalkan kerusakan dari alat serangan otomatis.

  • Pengidentifikasi sesi stateful harus dibatalkan di server setelah logout. Token JWT tanpa status harus berumur pendek sehingga jendela peluang bagi penyerang diminimalkan. Untuk JWT yang berumur lebih panjang, sangat disarankan untuk mengikuti standar OAuth untuk mencabut akses.

Ketentuan CodeIgniter

  • Folder Public, dengan aplikasi dan sistem di luar

  • Library Validation

  • Library Security menyediakan perlindungan CSRF

  • Library Session Library

  • Throttler untuk pembatasan tingkat

  • Filter Cross-Origin Resource Sharing (CORS)

  • Fungsi log_message() untuk logging

  • Framework otentikasi dan otorisasi resmi CodeIgniter Shield

  • Mudah menambahkan otentikasi pihak ketiga

A02:2021 Kegagalan Kriptografi

Hal pertama adalah menentukan kebutuhan perlindungan data dalam transit dan saat diam. Misalnya, kata sandi, nomor kartu kredit, catatan kesehatan, informasi pribadi, dan rahasia bisnis memerlukan perlindungan ekstra, terutama jika data tersebut berada di bawah undang-undang privasi, mis. Peraturan Perlindungan Data Umum (GDPR) Uni Eropa, atau peraturan, mis. perlindungan data keuangan seperti Standar Keamanan Data PCI (PCI DSS). Untuk semua data tersebut:

  • Apakah ada data yang ditransmisikan dalam teks biasa? Ini menyangkut protokol seperti HTTP, SMTP, FTP juga menggunakan peningkatan TLS seperti STARTTLS. Lalu lintas internet eksternal sangat berbahaya. Verifikasi semua lalu lintas internal, mis. antara penyeimbang beban, server web, atau sistem back-end.

  • Apakah ada algoritma atau protokol kriptografi lama atau lemah yang digunakan baik secara default atau dalam kode lama?

  • Apakah kunci kripto default digunakan, kunci kripto lemah dihasilkan atau digunakan kembali, atau apakah manajemen atau rotasi kunci yang tepat hilang? Apakah kunci kripto diperiksa ke repositori kode sumber?

  • Apakah enkripsi tidak diberlakukan, mis. apakah ada header HTTP (browser) direktif keamanan atau header yang hilang?

  • Apakah sertifikat server yang diterima dan rantai kepercayaan divalidasi dengan benar?

  • Apakah vektor inisialisasi diabaikan, digunakan kembali, atau tidak dihasilkan cukup aman untuk mode operasi kriptografi? Apakah mode operasi yang tidak aman seperti ECB digunakan? Apakah enkripsi digunakan ketika enkripsi terautentikasi lebih tepat?

  • Apakah kata sandi digunakan sebagai kunci kriptografi tanpa fungsi derivasi kunci berbasis kata sandi?

  • Apakah keacakan digunakan untuk tujuan kriptografi yang tidak dirancang untuk memenuhi persyaratan kriptografi? Bahkan jika fungsi yang benar dipilih, apakah perlu disemai oleh pengembang, dan jika tidak, apakah pengembang telah menimpa fungsionalitas penyemaian yang kuat yang dibangun di dalamnya dengan benih yang tidak memiliki entropi/ketidakpastian yang cukup?

  • Apakah fungsi hash yang tidak digunakan lagi seperti MD5 atau SHA1 digunakan, atau apakah fungsi hash non-kriptografi digunakan ketika fungsi hash kriptografi dibutuhkan?

  • Apakah metode padding kriptografi yang tidak digunakan lagi seperti PKCS nomor 1 v1.5 digunakan?

  • Apakah pesan kesalahan kriptografi atau informasi saluran samping dapat dieksploitasi, misalnya dalam bentuk serangan oracle padding?

Rekomendasi OWASP

Lakukan hal berikut, minimal, dan konsultasikan referensi:

  • Klasifikasikan data yang diproses, disimpan, atau ditransmisikan oleh aplikasi. Identifikasi data mana yang sensitif menurut undang-undang privasi, persyaratan peraturan, atau kebutuhan bisnis.

  • Jangan menyimpan data sensitif yang tidak perlu. Buang sesegera mungkin atau gunakan tokenisasi yang sesuai dengan PCI DSS atau bahkan pemotongan. Data yang tidak disimpan tidak dapat dicuri.

  • Pastikan untuk mengenkripsi semua data sensitif saat diam.

  • Pastikan algoritma, protokol, dan kunci standar yang mutakhir dan kuat ada; gunakan manajemen kunci yang tepat.

  • Enkripsi semua data dalam transit dengan protokol aman seperti TLS dengan cipher forward secrecy (FS), prioritas cipher oleh server, dan parameter aman. Terapkan enkripsi menggunakan direktif seperti HTTP Strict Transport Security (HSTS).

  • Nonaktifkan caching untuk respons yang berisi data sensitif.

  • Terapkan kontrol keamanan yang diperlukan sesuai klasifikasi data.

  • Jangan gunakan protokol lama seperti FTP dan SMTP untuk mengangkut data sensitif.

  • Simpan kata sandi menggunakan fungsi hashing yang kuat, adaptif, dan diberi salt dengan faktor kerja (faktor penundaan), seperti Argon2, scrypt, bcrypt, atau PBKDF2.

  • Vektor inisialisasi harus dipilih sesuai untuk mode operasi. Untuk banyak mode, ini berarti menggunakan CSPRNG (penghasil angka pseudo-acak yang aman secara kriptografi). Untuk mode yang membutuhkan nonce, maka vektor inisialisasi (IV) tidak memerlukan CSPRNG. Dalam semua kasus, IV tidak boleh digunakan dua kali untuk kunci tetap.

  • Selalu gunakan enkripsi terautentikasi daripada hanya enkripsi.

  • Kunci harus dihasilkan secara acak secara kriptografi dan disimpan dalam memori sebagai array byte. Jika kata sandi digunakan, maka harus dikonversi ke kunci melalui fungsi derivasi kunci berbasis kata sandi yang sesuai.

  • Pastikan keacakan kriptografi digunakan jika sesuai, dan bahwa itu belum disemai dengan cara yang dapat diprediksi atau dengan entropi rendah. Sebagian besar API modern tidak memerlukan pengembang untuk menyemai CSPRNG untuk mendapatkan keamanan.

  • Hindari fungsi kriptografi dan skema padding yang tidak digunakan lagi, seperti MD5, SHA1, PKCS nomor 1 v1.5.

  • Verifikasi secara independen efektivitas konfigurasi dan pengaturan.

Ketentuan CodeIgniter

  • Konfigurasi untuk akses aman global (Config\App::$forceGlobalSecureRequests)

  • Fungsi force_https()

  • Encryption Service

  • Konfigurasi database (encrypt)

  • Framework otentikasi dan otorisasi resmi CodeIgniter Shield

A03:2021 Injeksi

Aplikasi rentan terhadap serangan ketika:

  • Data yang disediakan pengguna tidak divalidasi, difilter, atau disanitasi oleh aplikasi.

  • Kueri dinamis atau panggilan non-parameter tanpa escaping yang sadar konteks digunakan langsung di interpreter.

  • Data yang tidak bersahabat digunakan dalam parameter pencarian Object-Relational Mapping (ORM) untuk mengekstrak catatan tambahan yang sensitif.

  • Data yang tidak bersahabat langsung digunakan atau digabungkan. SQL atau perintah berisi struktur dan data berbahaya dalam kueri dinamis, perintah, atau prosedur tersimpan.

Beberapa injeksi yang lebih umum adalah injeksi SQL, NoSQL, perintah OS, Object Relational Mapping (ORM), LDAP, dan Expression Language (EL) atau Object Graph Navigation Library (OGNL). Konsepnya identik di antara semua interpreter. Tinjauan kode sumber adalah metode terbaik untuk mendeteksi apakah aplikasi rentan terhadap injeksi. Pengujian otomatis semua parameter, header, URL, cookie, JSON, SOAP, dan input data XML sangat dianjurkan. Organisasi dapat menyertakan alat pengujian keamanan aplikasi statis (SAST), dinamis (DAST), dan interaktif (IAST) ke dalam pipeline CI/CD untuk mengidentifikasi cacat injeksi yang diperkenalkan sebelum deployment produksi.

Rekomendasi OWASP

Mencegah injeksi memerlukan pemisahan data dari perintah dan kueri:

  • Opsi yang disukai adalah menggunakan API yang aman, yang menghindari penggunaan interpreter sepenuhnya, menyediakan antarmuka berparameter, atau bermigrasi ke Alat Object Relational Mapping (ORM).

    • Catatan: Bahkan ketika berparameter, prosedur tersimpan masih dapat memperkenalkan injeksi SQL jika PL/SQL atau T-SQL menggabungkan kueri dan data atau mengeksekusi data yang tidak bersahabat dengan EXECUTE IMMEDIATE atau exec().

  • Gunakan validasi input sisi server positif. Ini bukan pertahanan lengkap karena banyak aplikasi memerlukan karakter khusus, seperti area teks atau API untuk aplikasi seluler.

  • Untuk kueri dinamis yang tersisa, escape karakter khusus menggunakan sintaks escape spesifik untuk interpreter tersebut.

    • Catatan: Struktur SQL seperti nama tabel, nama kolom, dan sebagainya tidak dapat di-escape, sehingga nama struktur yang disediakan pengguna berbahaya. Ini adalah masalah umum dalam perangkat lunak penulisan laporan.

  • Gunakan LIMIT dan kontrol SQL lainnya dalam kueri untuk mencegah pengungkapan massal catatan dalam kasus injeksi SQL.

Ketentuan CodeIgniter

  • URI Security

  • Filter InvalidChars

  • Library Validation

  • Fungsi esc()

  • Library HTTP menyediakan pemfilteran bidang input

  • Dukungan untuk Content Security Policy (CSP)

  • Class Query Builder

  • Metode escape database

  • Binding Kueri

A04:2021 Desain Tidak Aman

Desain yang tidak aman adalah kategori luas yang mewakili kelemahan yang berbeda, yang dinyatakan sebagai "desain kontrol yang hilang atau tidak efektif." Desain yang tidak aman bukanlah sumber untuk semua kategori risiko Top 10 lainnya. Ada perbedaan antara desain yang tidak aman dan implementasi yang tidak aman. Kami membedakan antara cacat desain dan cacat implementasi karena suatu alasan, mereka memiliki akar penyebab dan remediasi yang berbeda.

Desain yang aman masih dapat memiliki cacat implementasi yang mengarah pada kerentanan yang dapat dieksploitasi. Desain yang tidak aman tidak dapat diperbaiki dengan implementasi yang sempurna karena menurut definisi, kontrol keamanan yang dibutuhkan tidak pernah dibuat untuk bertahan dari serangan tertentu. Salah satu faktor yang berkontribusi pada desain yang tidak aman adalah kurangnya profiling risiko bisnis yang melekat pada perangkat lunak atau sistem yang sedang dikembangkan, dan dengan demikian kegagalan untuk menentukan tingkat desain keamanan yang diperlukan.

Rekomendasi OWASP

  • Tetapkan dan gunakan siklus hidup pengembangan yang aman dengan profesional AppSec untuk membantu mengevaluasi dan merancang kontrol terkait keamanan dan privasi

  • Tetapkan dan gunakan perpustakaan pola desain aman atau komponen siap pakai

  • Gunakan pemodelan ancaman untuk otentikasi kritis, kontrol akses, logika bisnis, dan alur kunci

  • Integrasikan bahasa dan kontrol keamanan ke dalam user story

  • Integrasikan pemeriksaan kelayakan di setiap tingkat aplikasi Anda (dari frontend hingga backend)

  • Tulis pengujian unit dan integrasi untuk memvalidasi bahwa semua alur kritis tahan terhadap model ancaman. Kompilasi kasus penggunaan dan kasus penyalahgunaan untuk setiap tingkat aplikasi Anda.

  • Pisahkan lapisan tingkat pada lapisan sistem dan jaringan tergantung pada paparan dan kebutuhan perlindungan

  • Pisahkan penyewa secara kuat berdasarkan desain di seluruh tingkatan

  • Batasi konsumsi sumber daya oleh pengguna atau layanan

Ketentuan CodeIgniter

  • Pengujian PHPUnit

  • Throttler untuk pembatasan tingkat

  • Framework otentikasi dan otorisasi resmi CodeIgniter Shield

A05:2021 Kesalahan Konfigurasi Keamanan

Aplikasi mungkin rentan jika aplikasi:

  • Kehilangan pengerasan keamanan yang sesuai di setiap bagian tumpukan aplikasi atau izin yang salah dikonfigurasi pada layanan cloud.

  • Fitur yang tidak perlu diaktifkan atau diinstal (misalnya, port, layanan, halaman, akun, atau hak istimewa yang tidak perlu).

  • Akun default dan kata sandi mereka masih diaktifkan dan tidak diubah.

  • Penanganan kesalahan mengungkapkan jejak tumpukan atau pesan kesalahan yang terlalu informatif lainnya kepada pengguna.

  • Untuk sistem yang ditingkatkan, fitur keamanan terbaru dinonaktifkan atau tidak dikonfigurasi dengan aman.

  • Pengaturan keamanan di server aplikasi, framework aplikasi (misalnya, Struts, Spring, ASP.NET), library, database, dll., tidak diatur ke nilai yang aman.

  • Server tidak mengirim header atau direktif keamanan, atau tidak diatur ke nilai yang aman.

  • Perangkat lunak sudah usang atau rentan (lihat A06:2021-Komponen Rentan dan Usang).

Tanpa proses konfigurasi keamanan aplikasi yang terkoordinasi dan berulang, sistem berisiko lebih tinggi.

Rekomendasi OWASP

Proses instalasi yang aman harus diterapkan, termasuk:

  • Proses pengerasan yang berulang membuatnya cepat dan mudah untuk menerapkan lingkungan lain yang terkunci dengan benar. Lingkungan pengembangan, QA, dan produksi harus dikonfigurasi secara identik, dengan kredensial yang berbeda digunakan di setiap lingkungan. Proses ini harus diotomatisasi untuk meminimalkan upaya yang diperlukan untuk menyiapkan lingkungan aman yang baru.

  • Platform minimal tanpa fitur, komponen, dokumentasi, dan sampel yang tidak perlu. Hapus atau jangan instal fitur dan framework yang tidak digunakan.

  • Tugas untuk meninjau dan memperbarui konfigurasi yang sesuai dengan semua catatan keamanan, pembaruan, dan patch sebagai bagian dari proses manajemen patch (lihat A06:2021-Komponen Rentan dan Usang). Tinjau izin penyimpanan cloud (misalnya, izin bucket S3).

  • Arsitektur aplikasi yang tersegmentasi menyediakan pemisahan yang efektif dan aman antara komponen atau penyewa, dengan segmentasi, containerization, atau grup keamanan cloud (ACL).

  • Mengirim direktif keamanan ke klien, mis. Header Keamanan.

  • Proses otomatis untuk memverifikasi efektivitas konfigurasi dan pengaturan di semua lingkungan.

Ketentuan CodeIgniter

  • Perintah spark config:check

  • Perintah spark phpini:check

  • Mode Produksi secara default

  • Filter SecureHeaders

A06:2021 Komponen Rentan dan Usang

Anda kemungkinan rentan:

  • Jika Anda tidak mengetahui versi semua komponen yang Anda gunakan (baik sisi klien maupun sisi server). Ini termasuk komponen yang Anda gunakan secara langsung serta dependensi bersarang.

  • Jika perangkat lunak rentan, tidak didukung, atau sudah usang. Ini termasuk OS, server web/aplikasi, sistem manajemen database (DBMS), aplikasi, API dan semua komponen, lingkungan runtime, dan library.

  • Jika Anda tidak memindai kerentanan secara teratur dan berlangganan buletin keamanan terkait komponen yang Anda gunakan.

  • Jika Anda tidak memperbaiki atau meningkatkan platform, framework, dan dependensi yang mendasari secara berbasis risiko, tepat waktu. Ini umumnya terjadi di lingkungan ketika patching adalah tugas bulanan atau triwulanan di bawah kontrol perubahan, membuat organisasi terbuka terhadap hari atau bulan paparan yang tidak perlu terhadap kerentanan yang telah diperbaiki.

  • Jika pengembang perangkat lunak tidak menguji kompatibilitas library yang diperbarui, ditingkatkan, atau dipatch.

  • Jika Anda tidak mengamankan konfigurasi komponen (lihat A05:2021-Kesalahan Konfigurasi Keamanan).

Rekomendasi OWASP

Harus ada proses manajemen patch yang berlaku untuk:

  • Hapus dependensi yang tidak digunakan, fitur yang tidak perlu, komponen, file, dan dokumentasi.

  • Terus-menerus menginventarisasi versi komponen sisi klien dan sisi server (misalnya, framework, library) dan dependensi mereka menggunakan alat seperti versi, OWASP Dependency Check, retire.js, dll. Terus-menerus memantau sumber seperti Common Vulnerability and Exposures (CVE) dan National Vulnerability Database (NVD) untuk kerentanan dalam komponen. Gunakan alat analisis komposisi perangkat lunak untuk mengotomatisasi proses. Berlangganan peringatan email untuk kerentanan keamanan terkait komponen yang Anda gunakan.

  • Hanya dapatkan komponen dari sumber resmi melalui tautan aman. Lebih suka paket yang ditandatangani untuk mengurangi kemungkinan menyertakan komponen yang dimodifikasi, berbahaya (Lihat A08:2021-Kegagalan Integritas Perangkat Lunak dan Data).

  • Pantau library dan komponen yang tidak terpelihara atau tidak membuat patch keamanan untuk versi lama. Jika patching tidak memungkinkan, pertimbangkan untuk menerapkan patch virtual untuk memantau, mendeteksi, atau melindungi dari masalah yang ditemukan.

Setiap organisasi harus memastikan rencana berkelanjutan untuk memantau, meninjau, dan menerapkan pembaruan atau perubahan konfigurasi selama masa pakai aplikasi atau portofolio.

A07:2021 Kegagalan Identifikasi dan Otentikasi

Konfirmasi identitas pengguna, otentikasi, dan manajemen sesi sangat penting untuk melindungi dari serangan terkait otentikasi. Mungkin ada kelemahan otentikasi jika aplikasi:

  • Mengizinkan serangan otomatis seperti credential stuffing, di mana penyerang memiliki daftar nama pengguna dan kata sandi yang valid.

  • Mengizinkan brute force atau serangan otomatis lainnya.

  • Mengizinkan kata sandi default, lemah, atau terkenal, seperti "Password1" atau "admin/admin".

  • Menggunakan proses pemulihan kredensial dan lupa kata sandi yang lemah atau tidak efektif, seperti "jawaban berbasis pengetahuan," yang tidak dapat dibuat aman.

  • Menggunakan teks biasa, terenkripsi, atau penyimpanan data kata sandi yang di-hash secara lemah (lihat A02:2021-Kegagalan Kriptografi).

  • Memiliki otentikasi multi-faktor yang hilang atau tidak efektif.

  • Mengekspos pengidentifikasi sesi di URL.

  • Menggunakan kembali pengidentifikasi sesi setelah login berhasil.

  • Tidak membatalkan ID Sesi dengan benar. Sesi pengguna atau token otentikasi (terutama token single sign-on (SSO)) tidak dibatalkan dengan benar selama logout atau periode tidak aktif.

Rekomendasi OWASP

  • Jika memungkinkan, terapkan otentikasi multi-faktor untuk mencegah serangan credential stuffing otomatis, brute force, dan serangan penggunaan kembali kredensial yang dicuri.

  • Jangan mengirim atau menerapkan dengan kredensial default apa pun, terutama untuk pengguna admin.

  • Terapkan pemeriksaan kata sandi lemah, seperti menguji kata sandi baru atau yang diubah terhadap daftar 10.000 kata sandi terburuk.

  • Sesuaikan panjang kata sandi, kompleksitas, dan kebijakan rotasi dengan pedoman National Institute of Standards and Technology (NIST) 800-63b di bagian 5.1.1 untuk Memorized Secrets atau kebijakan kata sandi modern berbasis bukti lainnya.

  • Pastikan pendaftaran, pemulihan kredensial, dan jalur API diperkuat terhadap serangan enumerasi akun dengan menggunakan pesan yang sama untuk semua hasil.

  • Batasi atau tunda secara bertahap upaya login yang gagal, tetapi berhati-hatilah agar tidak membuat skenario penolakan layanan. Catat semua kegagalan dan peringatkan administrator ketika credential stuffing, brute force, atau serangan lain terdeteksi.

  • Gunakan manajer sesi sisi server yang aman dan bawaan yang menghasilkan ID sesi acak baru dengan entropi tinggi setelah login. Pengidentifikasi sesi tidak boleh ada di URL, harus disimpan dengan aman, dan dibatalkan setelah logout, idle, dan batas waktu mutlak.

A08:2021 Kegagalan Integritas Perangkat Lunak dan Data

Kegagalan integritas perangkat lunak dan data berkaitan dengan kode dan infrastruktur yang tidak melindungi terhadap pelanggaran integritas. Contohnya adalah di mana aplikasi bergantung pada plugin, library, atau modul dari sumber yang tidak tepercaya, repositori, dan jaringan pengiriman konten (CDN). Pipeline CI/CD yang tidak aman dapat memperkenalkan potensi akses tidak sah, kode berbahaya, atau kompromi sistem.

Terakhir, banyak aplikasi sekarang menyertakan fungsionalitas pembaruan otomatis, di mana pembaruan diunduh tanpa verifikasi integritas yang memadai dan diterapkan ke aplikasi yang sebelumnya dipercaya. Penyerang berpotensi mengunggah pembaruan mereka sendiri untuk didistribusikan dan dijalankan di semua instalasi.

Contoh lain adalah di mana objek atau data dikodekan atau diserialkan ke dalam struktur yang dapat dilihat dan dimodifikasi oleh penyerang rentan terhadap deserialisasi yang tidak aman.

Rekomendasi OWASP

  • Gunakan tanda tangan digital atau mekanisme serupa untuk memverifikasi perangkat lunak atau data berasal dari sumber yang diharapkan dan belum diubah.

  • Pastikan library dan dependensi, seperti npm atau Maven, mengonsumsi repositori tepercaya. Jika Anda memiliki profil risiko yang lebih tinggi, pertimbangkan untuk menghosting repositori internal yang diketahui baik dan telah diverifikasi.

  • Pastikan alat keamanan rantai pasokan perangkat lunak, seperti OWASP Dependency Check atau OWASP CycloneDX, digunakan untuk memverifikasi bahwa komponen tidak mengandung kerentanan yang diketahui.

  • Pastikan ada proses peninjauan untuk perubahan kode dan konfigurasi untuk meminimalkan kemungkinan kode atau konfigurasi berbahaya dapat diperkenalkan ke dalam pipeline perangkat lunak Anda.

  • Pastikan pipeline CI/CD Anda memiliki pemisahan, konfigurasi, dan kontrol akses yang tepat untuk memastikan integritas kode yang mengalir melalui proses pembuatan dan deployment.

  • Pastikan data serial yang tidak ditandatangani atau tidak terenkripsi tidak dikirim ke klien yang tidak tepercaya tanpa beberapa bentuk pemeriksaan integritas atau tanda tangan digital untuk mendeteksi perusakan atau pemutaran ulang data serial.

A09:2021 Kegagalan Logging dan Pemantauan Keamanan

Kategori ini adalah untuk membantu mendeteksi, meningkatkan, dan menanggapi pelanggaran aktif. Tanpa logging dan pemantauan, pelanggaran tidak dapat dideteksi. Logging, deteksi, pemantauan, dan respons aktif yang tidak memadai terjadi kapan saja:

  • Peristiwa yang dapat diaudit, seperti login, login gagal, dan transaksi bernilai tinggi, tidak dicatat.

  • Peringatan dan kesalahan tidak menghasilkan pesan log yang memadai atau tidak jelas.

  • Log aplikasi dan API tidak dipantau untuk aktivitas mencurigakan.

  • Log hanya disimpan secara lokal.

  • Ambang batas peringatan dan proses eskalasi respons yang sesuai tidak ada atau tidak efektif.

  • Pengujian penetrasi dan pemindaian oleh alat pengujian keamanan aplikasi dinamis (DAST) (seperti OWASP ZAP) tidak memicu peringatan.

  • Aplikasi tidak dapat mendeteksi, meningkatkan, atau memperingatkan serangan aktif secara real-time atau mendekati real-time.

Anda rentan terhadap kebocoran informasi dengan membuat peristiwa logging dan peringatan terlihat oleh pengguna atau penyerang (lihat A01:2021-Kontrol Akses Rusak).

Rekomendasi OWASP

Pengembang harus menerapkan beberapa atau semua kontrol berikut, tergantung pada risiko aplikasi:

  • Pastikan semua kegagalan login, kontrol akses, dan validasi input sisi server dapat dicatat dengan konteks pengguna yang cukup untuk mengidentifikasi akun yang mencurigakan atau berbahaya dan disimpan cukup lama untuk memungkinkan analisis forensik yang tertunda.

  • Pastikan log dihasilkan dalam format yang mudah dikonsumsi oleh solusi manajemen log.

  • Pastikan data log dikodekan dengan benar untuk mencegah injeksi atau serangan pada sistem logging atau pemantauan.

  • Pastikan transaksi bernilai tinggi memiliki jejak audit dengan kontrol integritas untuk mencegah perusakan atau penghapusan, seperti tabel database hanya-tambah atau yang serupa.

  • Tim DevSecOps harus menetapkan pemantauan dan peringatan yang efektif sehingga aktivitas mencurigakan terdeteksi dan ditanggapi dengan cepat.

  • Menetapkan atau mengadopsi rencana respons insiden dan pemulihan, seperti National Institute of Standards and Technology (NIST) 800-61r2 atau yang lebih baru.

Ada framework perlindungan aplikasi komersial dan open-source seperti OWASP ModSecurity Core Rule Set, dan perangkat lunak korelasi log open-source, seperti stack Elasticsearch, Logstash, Kibana (ELK), yang menampilkan dashboard kustom dan peringatan.

A10:2021 Server-Side Request Forgery (SSRF)

Cacat SSRF terjadi setiap kali aplikasi web mengambil sumber daya jarak jauh tanpa memvalidasi URL yang disediakan pengguna. Ini memungkinkan penyerang untuk memaksa aplikasi mengirim permintaan yang dibuat ke tujuan yang tidak terduga, bahkan ketika dilindungi oleh firewall, VPN, atau jenis daftar kontrol akses jaringan (ACL) lainnya.

Karena aplikasi web modern menyediakan fitur yang nyaman bagi pengguna akhir, mengambil URL menjadi skenario umum. Akibatnya, insiden SSRF meningkat. Juga, tingkat keparahan SSRF semakin tinggi karena layanan cloud dan kompleksitas arsitektur.

Rekomendasi OWASP

Pengembang dapat mencegah SSRF dengan menerapkan beberapa atau semua kontrol pertahanan mendalam berikut:

Dari lapisan Jaringan:

  • Segmentasikan fungsionalitas akses sumber daya jarak jauh di jaringan terpisah untuk mengurangi dampak SSRF

  • Terapkan kebijakan firewall "tolak secara default" atau aturan kontrol akses jaringan untuk memblokir semua kecuali lalu lintas intranet penting.

    • Petunjuk:

      • Tetapkan kepemilikan dan siklus hidup untuk aturan firewall berdasarkan aplikasi.

      • Catat semua aliran jaringan yang diterima dan diblokir pada firewall (lihat A09:2021-Kegagalan Logging dan Pemantauan Keamanan).

Dari lapisan Aplikasi:

  • Sanitasi dan validasi semua data input yang disediakan klien

  • Terapkan skema URL, port, dan tujuan dengan daftar izin positif

  • Jangan mengirim respons mentah ke klien

  • Nonaktifkan pengalihan HTTP

  • Sadari konsistensi URL untuk menghindari serangan seperti DNS rebinding dan kondisi balapan "waktu pemeriksaan, waktu penggunaan" (TOCTOU)

Jangan mitigasi SSRF melalui penggunaan daftar tolak atau ekspresi reguler. Penyerang memiliki daftar payload, alat, dan keterampilan untuk melewati daftar tolak.

Ketentuan CodeIgniter

  • Library Validation

  • Library HTTP menyediakan pemfilteran bidang input

OWASP API Security Top 10 2023

API1:2023 Otorisasi Tingkat Objek Rusak

API cenderung mengekspos endpoint yang menangani pengidentifikasi objek, menciptakan luasnya permukaan serangan masalah Kontrol Akses Tingkat Objek. Pemeriksaan otorisasi tingkat objek harus dipertimbangkan di setiap fungsi yang mengakses sumber data menggunakan ID dari pengguna.

Rekomendasi OWASP

  • Terapkan mekanisme otorisasi yang tepat yang bergantung pada kebijakan pengguna dan hierarki.

  • Gunakan mekanisme otorisasi untuk memeriksa apakah pengguna yang masuk memiliki akses untuk melakukan tindakan yang diminta pada catatan di setiap fungsi yang menggunakan input dari klien untuk mengakses catatan di database.

  • Lebih suka menggunakan nilai acak dan tidak dapat diprediksi sebagai GUID untuk ID catatan.

  • Tulis pengujian untuk mengevaluasi kerentanan mekanisme otorisasi. Jangan menerapkan perubahan yang menyebabkan pengujian gagal.

Ketentuan CodeIgniter

  • Framework otentikasi dan otorisasi resmi CodeIgniter Shield

  • Pengujian PHPUnit

API2:2023 Otentikasi Rusak

Mekanisme otentikasi sering diimplementasikan dengan tidak benar, memungkinkan penyerang untuk mengkompromikan token otentikasi atau untuk mengeksploitasi cacat implementasi untuk mengambil identitas pengguna lain sementara atau permanen. Mengkompromikan kemampuan sistem untuk mengidentifikasi klien/pengguna, mengkompromikan keamanan API secara keseluruhan.

Rekomendasi OWASP

  • Pastikan Anda mengetahui semua alur yang mungkin untuk otentikasi ke API (seluler/ web/deep links yang mengimplementasikan otentikasi satu klik/dll.). Tanyakan kepada engineer Anda alur apa yang Anda lewatkan.

  • Baca tentang mekanisme otentikasi Anda. Pastikan Anda memahami apa dan bagaimana mereka digunakan. OAuth bukanlah otentikasi, begitu juga kunci API.

  • Jangan menciptakan kembali roda dalam otentikasi, pembuatan token, atau penyimpanan kata sandi. Gunakan standar.

  • Endpoint pemulihan kredensial/lupa kata sandi harus diperlakukan sebagai endpoint login dalam hal brute force, pembatasan tingkat, dan perlindungan penguncian.

  • Membutuhkan otentikasi ulang untuk operasi sensitif (misalnya mengubah email pemilik akun/nomor telepon 2FA).

  • Gunakan OWASP Authentication Cheatsheet.

  • Jika memungkinkan, terapkan otentikasi multi-faktor.

  • Terapkan mekanisme anti-brute force untuk mengurangi credential stuffing, serangan kamus, dan serangan brute force pada endpoint otentikasi Anda. Mekanisme ini harus lebih ketat daripada mekanisme pembatasan tingkat reguler pada API Anda.

  • Terapkan mekanisme penguncian akun/captcha untuk mencegah serangan brute force terhadap pengguna tertentu. Terapkan pemeriksaan kata sandi lemah.

  • Kunci API tidak boleh digunakan untuk otentikasi pengguna. Mereka hanya boleh digunakan untuk otentikasi klien API.

Ketentuan CodeIgniter

  • Controller Filters

  • Perintah spark routes

  • Framework otentikasi dan otorisasi resmi CodeIgniter Shield

  • Throttler untuk pembatasan tingkat

API3:2023 Otorisasi Tingkat Properti Objek Rusak

Kategori ini menggabungkan API3:2019 Excessive Data Exposure dan API6:2019 - Mass Assignment, berfokus pada akar penyebab: kurangnya atau validasi otorisasi yang tidak tepat pada tingkat properti objek. Ini mengarah pada pengungkapan informasi atau manipulasi oleh pihak yang tidak berwenang.

Rekomendasi OWASP

  • Saat mengekspos objek menggunakan endpoint API, selalu pastikan bahwa pengguna harus memiliki akses ke properti objek yang Anda ekspos.

  • Hindari menggunakan metode generik seperti to_json() dan to_string(). Sebaliknya, pilih properti objek tertentu yang ingin Anda kembalikan.

  • Jika memungkinkan, hindari menggunakan fungsi yang secara otomatis mengikat input klien ke dalam variabel kode, objek internal, atau properti objek ("Mass Assignment").

  • Izinkan perubahan hanya pada properti objek yang harus diperbarui oleh klien.

  • Terapkan mekanisme validasi respons berbasis skema sebagai lapisan keamanan tambahan. Sebagai bagian dari mekanisme ini, definisikan dan terapkan data yang dikembalikan oleh semua metode API.

  • Pertahankan struktur data yang dikembalikan seminimal mungkin, sesuai dengan persyaratan bisnis/fungsional untuk endpoint.

Ketentuan CodeIgniter

  • Model $allowedFields

  • Framework otentikasi dan otorisasi resmi CodeIgniter Shield

API4:2023 Konsumsi Sumber Daya Tidak Terbatas

Memenuhi permintaan API membutuhkan sumber daya seperti bandwidth jaringan, CPU, memori, dan penyimpanan. Sumber daya lain seperti email/SMS/panggilan telepon atau validasi biometrik disediakan oleh penyedia layanan melalui integrasi API, dan dibayar per permintaan. Serangan yang berhasil dapat menyebabkan Penolakan Layanan atau peningkatan biaya operasional.

Rekomendasi OWASP

  • Gunakan solusi yang memudahkan pembatasan memori, CPU, jumlah restart, deskriptor file, dan proses seperti Kontainer / kode Tanpa Server (misalnya Lambda).

  • Tentukan dan terapkan ukuran maksimum data pada semua parameter dan payload yang masuk, seperti panjang maksimum untuk string, jumlah maksimum elemen dalam array, dan ukuran file unggahan maksimum (terlepas dari apakah disimpan secara lokal atau di penyimpanan cloud).

  • Terapkan batasan seberapa sering klien dapat berinteraksi dengan API dalam jangka waktu yang ditentukan (pembatasan tingkat).

  • Pembatasan tingkat harus disesuaikan berdasarkan kebutuhan bisnis. Beberapa Endpoint API mungkin memerlukan kebijakan yang lebih ketat.

  • Batasi/throttle berapa kali atau seberapa sering satu klien/pengguna API dapat mengeksekusi satu operasi (misalnya memvalidasi OTP, atau meminta pemulihan kata sandi tanpa mengunjungi URL satu kali).

  • Tambahkan validasi sisi server yang tepat untuk string kueri dan parameter badan permintaan, khususnya yang mengontrol jumlah catatan yang akan dikembalikan dalam respons.

  • Konfigurasi batas pengeluaran untuk semua penyedia layanan/integrasi API. Saat menetapkan batas pengeluaran tidak memungkinkan, peringatan penagihan harus dikonfigurasi sebagai gantinya.

Ketentuan CodeIgniter

  • Library Validation

  • Throttler untuk pembatasan tingkat

API5:2023 Otorisasi Tingkat Fungsi Rusak

Kebijakan kontrol akses yang kompleks dengan hierarki, grup, dan peran yang berbeda, dan pemisahan yang tidak jelas antara fungsi administratif dan reguler, cenderung mengarah pada cacat otorisasi. Dengan mengeksploitasi masalah ini, penyerang dapat memperoleh akses ke sumber daya pengguna lain dan/atau fungsi administratif.

Rekomendasi OWASP

Aplikasi Anda harus memiliki modul otorisasi yang konsisten dan mudah dianalisis yang dipanggil dari semua fungsi bisnis Anda. Seringkali, perlindungan tersebut disediakan oleh satu atau lebih komponen di luar kode aplikasi.

  • Mekanisme penegakan harus menolak semua akses secara default, membutuhkan izin eksplisit ke peran tertentu untuk akses ke setiap fungsi.

  • Tinjau endpoint API Anda terhadap cacat otorisasi tingkat fungsi, sambil mempertimbangkan logika bisnis aplikasi dan hierarki grup.

  • Pastikan bahwa semua controller administratif Anda mewarisi dari controller abstrak administratif yang mengimplementasikan pemeriksaan otorisasi berdasarkan grup/peran pengguna.

  • Pastikan bahwa fungsi administratif di dalam controller reguler mengimplementasikan pemeriksaan otorisasi berdasarkan grup dan peran pengguna.

Ketentuan CodeIgniter

  • Controller Filters

  • Framework otentikasi dan otorisasi resmi CodeIgniter Shield

API6:2023 Akses Tidak Terbatas ke Alur Bisnis Sensitif

API yang rentan terhadap risiko ini mengekspos alur bisnis - seperti membeli tiket, atau memposting komentar - tanpa mengkompensasi bagaimana fungsionalitas tersebut dapat membahayakan bisnis jika digunakan secara berlebihan secara otomatis. Ini tidak selalu berasal dari bug implementasi.

Rekomendasi OWASP

Perencanaan mitigasi harus dilakukan dalam dua lapisan:

  • Bisnis - identifikasi alur bisnis yang dapat membahayakan bisnis jika digunakan secara berlebihan.

  • Rekayasa - pilih mekanisme perlindungan yang tepat untuk mengurangi risiko bisnis.

Beberapa mekanisme perlindungan lebih sederhana sementara yang lain lebih sulit untuk diimplementasikan. Metode-metode berikut digunakan untuk memperlambat ancaman otomatis:

  • Sidik jari perangkat: menolak layanan ke perangkat klien yang tidak terduga (misalnya browser tanpa kepala) cenderung membuat aktor ancaman menggunakan solusi yang lebih canggih, sehingga lebih mahal bagi mereka

  • Deteksi manusia: menggunakan captcha atau solusi biometrik yang lebih canggih (misalnya pola mengetik)

  • Pola non-manusia: menganalisis alur pengguna untuk mendeteksi pola non-manusia (misalnya pengguna mengakses fungsi "tambah ke keranjang" dan "selesaikan pembelian" dalam kurang dari satu detik)

  • Pertimbangkan untuk memblokir alamat IP node keluar Tor dan proxy yang terkenal

Amankan dan batasi akses ke API yang dikonsumsi langsung oleh mesin (seperti API pengembang dan B2B). Mereka cenderung menjadi target mudah bagi penyerang karena mereka sering tidak mengimplementasikan semua mekanisme perlindungan yang diperlukan.

Ketentuan CodeIgniter

  • n/a

API7:2023 Server Side Request Forgery

Cacat Server-Side Request Forgery (SSRF) dapat terjadi ketika API mengambil sumber daya jarak jauh tanpa memvalidasi URI yang disediakan pengguna. Ini memungkinkan penyerang untuk memaksa aplikasi mengirim permintaan yang dibuat ke tujuan yang tidak terduga, bahkan ketika dilindungi oleh firewall atau VPN.

Rekomendasi OWASP

  • Isolasi mekanisme pengambilan sumber daya di jaringan Anda: biasanya fitur-fitur ini bertujuan untuk mengambil sumber daya jarak jauh dan bukan sumber daya internal.

  • Kapan pun memungkinkan, gunakan daftar izin untuk:

    • Asal jarak jauh yang diharapkan pengguna untuk mengunduh sumber daya dari (misalnya Google Drive, Gravatar, dll.)

    • Skema URL dan port

    • Tipe media yang diterima untuk fungsionalitas tertentu

  • Nonaktifkan pengalihan HTTP.

  • Gunakan parser URL yang teruji dengan baik dan terpelihara untuk menghindari masalah yang disebabkan oleh inkonsistensi parsing URL.

  • Validasi dan sanitasi semua data input yang disediakan klien.

  • Jangan mengirim respons mentah ke klien.

Ketentuan CodeIgniter

  • Library Validation

  • Library HTTP menyediakan pemfilteran bidang input

  • Class CURLRequest

  • Class URI

API8:2023 Kesalahan Konfigurasi Keamanan

API dan sistem pendukungnya biasanya berisi konfigurasi yang kompleks, dimaksudkan untuk membuat API lebih dapat disesuaikan. Insinyur perangkat lunak dan DevOps dapat melewatkan konfigurasi ini, atau tidak mengikuti praktik terbaik keamanan dalam hal konfigurasi, membuka pintu untuk berbagai jenis serangan.

Rekomendasi OWASP

Siklus hidup API harus mencakup:

  • Proses pengerasan yang berulang yang mengarah pada deployment lingkungan yang terkunci dengan benar yang cepat dan mudah

  • Tugas untuk meninjau dan memperbarui konfigurasi di seluruh tumpukan API. Peninjauan harus mencakup: file orkestrasi, komponen API, dan layanan cloud (misalnya izin bucket S3)

  • Proses otomatis untuk terus-menerus menilai efektivitas konfigurasi dan pengaturan di semua lingkungan

Selanjutnya:

  • Pastikan semua komunikasi API dari klien ke server API dan komponen downstream/upstream terjadi melalui saluran komunikasi terenkripsi (TLS), terlepas dari apakah itu API internal atau publik.

  • Bersikaplah spesifik tentang verb HTTP mana yang dapat diakses oleh setiap API: semua verb HTTP lainnya harus dinonaktifkan (misalnya HEAD).

  • API yang diharapkan diakses dari klien berbasis browser (misalnya, front-end WebApp) harus, setidaknya:

    • mengimplementasikan kebijakan Cross-Origin Resource Sharing (CORS) yang tepat

    • menyertakan Header Keamanan yang berlaku

  • Batasi jenis konten/format data yang masuk ke yang memenuhi persyaratan bisnis/fungsional.

  • Pastikan semua server dalam rantai server HTTP (misalnya penyeimbang beban, proxy terbalik dan maju, dan server back-end) memproses permintaan masuk secara seragam untuk menghindari masalah desync.

  • Jika berlaku, definisikan dan terapkan semua skema payload respons API, termasuk respons kesalahan, untuk mencegah jejak pengecualian dan informasi berharga lainnya dikirim kembali ke penyerang.

Ketentuan CodeIgniter

  • Konfigurasi untuk akses aman global (Config\App::$forceGlobalSecureRequests)

  • Fungsi force_https()

  • Defined Route Routing

  • Auto Routing (Improved)

  • Cross-Origin Resource Sharing (CORS) filter

API9:2023 Manajemen Inventaris yang Tidak Tepat

API cenderung mengekspos lebih banyak endpoint daripada aplikasi web tradisional, membuat dokumentasi yang tepat dan terbaru sangat penting. Inventaris host dan versi API yang di-deploy juga penting untuk mengurangi masalah seperti versi API yang tidak digunakan lagi dan endpoint debug yang terekspos.

Rekomendasi OWASP

  • Inventaris semua host API dan dokumentasikan aspek-aspek penting dari masing-masing, berfokus pada lingkungan API (misalnya produksi, staging, pengujian, pengembangan), siapa yang seharusnya memiliki akses jaringan ke host (misalnya publik, internal, mitra) dan versi API.

  • Inventaris layanan terintegrasi dan dokumentasikan aspek-aspek penting seperti peran mereka dalam sistem, data apa yang dipertukarkan (alur data), dan sensitivitasnya.

  • Dokumentasikan semua aspek API Anda seperti otentikasi, kesalahan, pengalihan, pembatasan tingkat, kebijakan cross-origin resource sharing (CORS), dan endpoint, termasuk parameter, permintaan, dan responsnya.

  • Hasilkan dokumentasi secara otomatis dengan mengadopsi standar terbuka. Sertakan pembuatan dokumentasi dalam pipeline CI/CD Anda.

  • Jadikan dokumentasi API hanya tersedia bagi mereka yang berwenang untuk menggunakan API.

  • Gunakan tindakan perlindungan eksternal seperti solusi khusus keamanan API untuk semua versi API Anda yang terekspos, tidak hanya untuk versi produksi saat ini.

  • Hindari menggunakan data produksi dengan deployment API non-produksi. Jika ini tidak dapat dihindari, endpoint ini harus mendapatkan perlakuan keamanan yang sama dengan yang produksi.

  • Ketika versi API yang lebih baru menyertakan peningkatan keamanan, lakukan analisis risiko untuk menginformasikan tindakan mitigasi yang diperlukan untuk versi yang lebih lama. Misalnya, apakah mungkin untuk mem-backport peningkatan tanpa merusak kompatibilitas API atau jika Anda perlu menghapus versi lama dengan cepat dan memaksa semua klien untuk beralih ke versi terbaru.

Ketentuan CodeIgniter

  • Perintah spark routes

API10:2023 Konsumsi API yang Tidak Aman

Pengembang cenderung lebih mempercayai data yang diterima dari API pihak ketiga daripada input pengguna, dan karenanya cenderung mengadopsi standar keamanan yang lebih lemah. Untuk mengkompromikan API, penyerang mengejar layanan pihak ketiga yang terintegrasi daripada mencoba mengkompromikan API target secara langsung.

Rekomendasi OWASP

  • Saat mengevaluasi penyedia layanan, nilai postur keamanan API mereka.

  • Pastikan semua interaksi API terjadi melalui saluran komunikasi yang aman (TLS).

  • Selalu validasi dan sanitasi data yang diterima dari API terintegrasi sebelum menggunakannya.

  • Pertahankan daftar izin lokasi terkenal yang mungkin dialihkan oleh API terintegrasi Anda: jangan mengikuti pengalihan secara membabi buta.

Ketentuan CodeIgniter

  • Class CURLRequest

  • Library Validation
