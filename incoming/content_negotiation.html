Apa itu Negosiasi Konten?

Negosiasi konten adalah cara untuk menentukan jenis konten apa yang akan dikembalikan ke klien berdasarkan apa yang dapat ditangani oleh klien, dan apa yang dapat ditangani oleh server. Ini dapat digunakan untuk menentukan apakah klien menginginkan HTML atau JSON yang dikembalikan, apakah gambar harus dikembalikan sebagai JPEG atau PNG, jenis kompresi apa yang didukung dan banyak lagi. Ini dilakukan dengan menganalisis empat header berbeda yang masing-masing dapat mendukung beberapa opsi nilai, masing-masing dengan prioritasnya sendiri.

Mencoba mencocokkan ini secara manual bisa sangat menantang. CodeIgniter menyediakan class Negotiator yang dapat menangani ini untuk Anda.

Pada intinya, Negosiasi Konten hanyalah bagian dari spesifikasi HTTP yang memungkinkan satu resource untuk menyajikan lebih dari satu jenis konten, memungkinkan klien untuk meminta jenis data yang paling sesuai untuk mereka.

Contoh klasik dari ini adalah browser yang tidak dapat menampilkan berkas PNG dapat meminta hanya gambar GIF atau JPEG. Ketika server menerima permintaan, ia melihat jenis berkas yang tersedia yang diminta klien dan memilih kecocokan terbaik dari format gambar yang didukungnya, dalam kasus ini kemungkinan memilih gambar JPEG untuk dikembalikan.

Negosiasi yang sama ini dapat terjadi dengan empat jenis data:

  • Jenis Media/Dokumen - ini bisa berupa format gambar, atau HTML vs. XML atau JSON.

  • Set Karakter - Set karakter yang harus diatur dalam dokumen yang dikembalikan. Biasanya adalah UTF-8.

  • Pengkodean Dokumen - Biasanya jenis kompresi yang digunakan pada hasil.

  • Bahasa Dokumen - Untuk situs yang mendukung banyak bahasa, ini membantu menentukan mana yang akan dikembalikan.

Memuat Class

Anda dapat memuat instance class secara manual melalui class Service:

<?php

$negotiate = service('negotiator');

Ini akan mengambil instance permintaan saat ini dan secara otomatis menyuntikkannya ke dalam class Negotiator.

Class ini tidak perlu dimuat sendiri. Sebaliknya, itu dapat diakses melalui instance IncomingRequest permintaan ini. Meskipun Anda tidak dapat mengaksesnya secara langsung dengan cara ini, Anda dapat dengan mudah mengakses semua method melalui method negotiate():

<?php

$request->negotiate('media', ['foo', 'bar']);

Ketika diakses dengan cara ini, parameter pertama adalah jenis konten yang Anda coba temukan kecocokannya, sedangkan yang kedua adalah array nilai yang didukung.

Negosiasi

Di bagian ini, kita akan membahas 4 jenis konten yang dapat dinegosiasikan dan menunjukkan bagaimana itu akan terlihat menggunakan kedua method yang dijelaskan di atas untuk mengakses negosiator.

Media

Aspek pertama yang perlu dilihat adalah penanganan negosiasi 'media'. Ini disediakan oleh header Accept dan merupakan salah satu header paling kompleks yang tersedia. Contoh umum adalah klien memberi tahu server format apa yang diinginkannya data. Ini sangat umum dalam API. Misalnya, klien mungkin meminta data berformat JSON dari endpoint API:

GET /foo HTTP/1.1
Accept: application/json

Server sekarang perlu menyediakan daftar jenis konten apa yang dapat disediakannya. Dalam contoh ini, API mungkin dapat mengembalikan data sebagai HTML mentah, JSON, atau XML. Daftar ini harus disediakan dalam urutan preferensi:

<?php

$supported = [
    'application/json',
    'text/html',
    'application/xml',
];

$format = $request->negotiate('media', $supported);
// or
$format = $negotiate->media($supported);

Dalam kasus ini, baik klien maupun server dapat menyetujui pemformatan data sebagai JSON sehingga 'json' dikembalikan dari method negosiasi. Secara default, jika tidak ada kecocokan yang ditemukan, elemen pertama dalam array $supported akan dikembalikan. Namun, dalam beberapa kasus, Anda mungkin perlu memberlakukan format agar sesuai persis. Jika Anda meneruskan true sebagai nilai akhir, itu akan mengembalikan string kosong jika tidak ada kecocokan yang ditemukan:

<?php

$format = $request->negotiate('media', $supported, true);
// or
$format = $negotiate->media($supported, true);

Bahasa

Penggunaan umum lainnya adalah untuk menentukan bahasa di mana konten harus disajikan. Jika Anda hanya menjalankan situs satu bahasa, ini jelas tidak akan membuat banyak perbedaan, tetapi situs mana pun yang dapat menawarkan beberapa terjemahan konten akan menemukan ini berguna, karena browser biasanya akan mengirim bahasa yang disukai di header Accept-Language

GET /foo HTTP/1.1
Accept-Language: fr; q=1.0, en; q=0.5

Dalam contoh ini, browser akan lebih memilih bahasa Prancis, dengan pilihan kedua bahasa Inggris. Jika situs web Anda mendukung bahasa Inggris dan Jerman, Anda akan melakukan sesuatu seperti:

<?php

$supported = [
    'en',
    'de',
];

$lang = $request->negotiate('language', $supported);
// or
$lang = $negotiate->language($supported);

Dalam contoh ini, 'en' akan dikembalikan sebagai bahasa saat ini. Jika tidak ada kecocokan yang ditemukan, itu akan mengembalikan elemen pertama dalam array $supported, sehingga itu harus selalu menjadi bahasa yang disukai.

Negosiasi Lokal yang Ketat

Baru pada versi 4.6.0.

Secara default, lokal ditentukan berdasarkan perbandingan yang hilang. Jadi hanya bagian pertama dari string lokal yang dipertimbangkan (bahasa). Ini biasanya cukup. Tetapi terkadang kita ingin dapat membedakan antara versi regional seperti en-US dan en-GB untuk menyajikan konten yang berbeda.

Untuk kasus seperti itu, kami telah memperkenalkan pengaturan baru yang dapat diaktifkan melalui Config\Feature::$strictLocaleNegotiation. Ini akan memastikan bahwa perbandingan ketat akan dilakukan terlebih dahulu.

Catatan

CodeIgniter dilengkapi dengan terjemahan hanya untuk tag bahasa utama ('en', 'fr', dll.). Jadi jika Anda mengaktifkan fitur ini dan pengaturan Anda di Config\App::$supportedLocales menyertakan tag bahasa regional ('en-US', 'fr-FR', dll.), maka perlu diingat bahwa jika Anda memiliki berkas terjemahan Anda sendiri, Anda juga harus mengubah nama folder untuk berkas terjemahan CodeIgniter agar sesuai dengan apa yang Anda masukkan dalam array $supportedLocales.

Sekarang mari kita pertimbangkan contoh di bawah ini. Bahasa pilihan browser akan diatur seperti ini:

GET /foo HTTP/1.1
Accept-Language: fr; q=1.0, en-GB; q=0.5

Dalam contoh ini, browser akan lebih memilih bahasa Prancis, dengan pilihan kedua bahasa Inggris (Britania Raya). Situs web Anda di sisi lain mendukung bahasa Jerman dan Inggris (Amerika Serikat):

<?php

$supported = [
    'de',
    'en-US',
];

$lang = $request->negotiate('language', $supported);
// or
$lang = $negotiate->language($supported);

Dalam contoh ini, 'en-US' akan dikembalikan sebagai bahasa saat ini. Jika tidak ada kecocokan yang ditemukan, itu akan mengembalikan elemen pertama dalam array $supported. Begini cara kerja proses pemilihan lokal.

Meskipun 'fr' lebih disukai oleh browser, itu tidak ada dalam array $supported kami. Masalah yang sama terjadi dengan 'en-GB', tetapi di sini kita akan dapat mencari varian. Pertama, kita akan kembali ke lokal yang paling umum (dalam kasus ini 'en') yang lagi-lagi tidak ada dalam array kami. Kemudian kita akan mencari lokal regional 'en-'. Dan saat itulah nilai kita dari array $supported akan dicocokkan. Kita akan mengembalikan 'en-US'.

Jadi proses pemilihan lokal adalah sebagai berikut:

  1. kecocokan ketat ('en-GB') - ISO 639-1 ditambah ISO 3166-1 alpha-2

  2. kecocokan lokal umum ('en') - ISO 639-1

  3. kecocokan lokal regional ('en-') - ISO 639-1 ditambah "wildcard" untuk ISO 3166-1 alpha-2

Pengkodean

Header Accept-Encoding berisi set karakter yang lebih disukai klien untuk diterima, dan digunakan untuk menentukan jenis kompresi yang didukung klien:

GET /foo HTTP/1.1
Accept-Encoding: compress, gzip

Web server Anda akan menentukan jenis kompresi apa yang dapat Anda gunakan. Beberapa, seperti Apache, hanya mendukung gzip:

<?php

$type = $request->negotiate('encoding', ['gzip']);
// or
$type = $negotiate->encoding(['gzip']);

Lihat lebih lanjut di Wikipedia.

Set Karakter

Set karakter yang diinginkan diteruskan melalui header Accept-Charset:

GET /foo HTTP/1.1
Accept-Charset: utf-16, utf-8

Secara default, jika tidak ada kecocokan yang ditemukan, utf-8 akan dikembalikan:

<?php

$charset = $request->negotiate('charset', ['utf-8']);
// or
$charset = $negotiate->charset(['utf-8']);
