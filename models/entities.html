Menggunakan Class Entitas

CodeIgniter mendukung class Entitas sebagai warga kelas satu, sambil tetap menjaga penggunaannya sepenuhnya opsional. Class ini umumnya digunakan sebagai bagian dari pola Repository, tetapi dapat digunakan secara langsung dengan Model jika itu lebih sesuai dengan kebutuhan Anda.

Penggunaan Entitas

Pada intinya, class Entitas hanyalah sebuah class yang merepresentasikan satu baris database. Class ini memiliki properti untuk merepresentasikan kolom database, dan menyediakan metode tambahan apa pun untuk mengimplementasikan logika bisnis untuk baris tersebut.

Catatan

Untuk kemudahan pemahaman, penjelasan di sini didasarkan pada kasus penggunaan database. Namun, Entitas juga dapat digunakan untuk data yang tidak berasal dari database.

Fitur intinya, bagaimanapun, adalah bahwa ia tidak tahu apa-apa tentang bagaimana cara mempertahankan dirinya sendiri. Itu adalah tanggung jawab dari model atau class repository. Dengan begitu, jika ada perubahan pada cara Anda perlu menyimpan objek, Anda tidak perlu mengubah cara objek tersebut digunakan di seluruh aplikasi.

Ini memungkinkan untuk menggunakan file JSON atau XML untuk menyimpan objek selama tahap prototipe cepat, dan kemudian dengan mudah beralih ke database ketika Anda telah membuktikan konsepnya berhasil.

Mari kita bahas Entitas Pengguna yang sangat sederhana dan bagaimana kita akan bekerja dengannya untuk membantu memperjelas.

Asumsikan Anda memiliki tabel database bernama users yang memiliki skema berikut:

id          - integer
username    - string
email       - string
password    - string
created_at  - datetime

Penting

attributes adalah kata yang dicadangkan untuk penggunaan internal. Jika Anda menggunakannya sebagai nama kolom, Entitas tidak berfungsi dengan benar.

Membuat Class Entitas

Sekarang buat class Entitas baru. Karena tidak ada lokasi default untuk menyimpan class-class ini, dan itu tidak cocok dengan struktur direktori yang ada, buat direktori baru di app/Entities. Buat Entitas itu sendiri di app/Entities/User.php.

<?php

namespace App\Entities;

use CodeIgniter\Entity\Entity;

class User extends Entity
{
    // ...
}

Paling sederhana, ini semua yang perlu Anda lakukan, meskipun kita akan membuatnya lebih berguna sebentar lagi.

Membuat Model

Buat model terlebih dahulu di app/Models/UserModel.php agar kita dapat berinteraksi dengannya:

<?php

namespace App\Models;

use CodeIgniter\Model;

class UserModel extends Model
{
    protected $table         = 'users';
    protected $allowedFields = [
        'username', 'email', 'password',
    ];
    protected $returnType    = \App\Entities\User::class;
    protected $useTimestamps = true;
}

Model ini menggunakan tabel users di database untuk semua aktivitasnya.

Kita telah mengatur properti $allowedFields untuk menyertakan semua field yang kita ingin class eksternal ubah. Field id, created_at, dan updated_at ditangani secara otomatis oleh class atau database, jadi kita tidak ingin mengubahnya.

Terakhir, kita telah mengatur class Entitas kita sebagai $returnType. Ini memastikan bahwa semua metode bawaan pada model yang mengembalikan baris dari database akan mengembalikan instance dari class Entitas Pengguna kita alih-alih objek atau array seperti biasa.

Catatan

Tentu saja, jika Anda menambahkan metode kustom ke model Anda, Anda harus mengimplementasikannya sehingga instance dari $returnType dikembalikan.

Bekerja dengan Class Entitas

Sekarang semua bagian sudah ada, Anda akan bekerja dengan class Entitas seperti Anda bekerja dengan class lainnya:

<?php

$user = $userModel->find($id);

// Display
echo $user->username;
echo $user->email;

// Updating
unset($user->username);

if (! isset($user->username)) {
    $user->username = 'something new';
}

$userModel->save($user);

// Create
$user           = new \App\Entities\User();
$user->username = 'foo';
$user->email    = 'foo@example.com';
$userModel->save($user);

Anda mungkin telah memperhatikan bahwa class User belum mengatur properti apa pun untuk kolom-kolomnya, tetapi Anda masih dapat mengaksesnya seolah-olah itu adalah properti publik. Class dasar, CodeIgniter\Entity\Entity, mengurus ini untuk Anda, serta menyediakan kemampuan untuk memeriksa properti dengan isset(), atau unset() properti, dan melacak kolom mana yang telah berubah sejak objek dibuat atau ditarik dari database.

Catatan

Class Entitas menyimpan data dalam properti class $attributes secara internal.

Ketika Pengguna diteruskan ke metode save() model, secara otomatis ia akan membaca properti dan menyimpan setiap perubahan pada kolom yang terdaftar dalam properti $allowedFields model. Ia juga tahu apakah akan membuat baris baru, atau memperbarui yang sudah ada.

Catatan

Ketika kita memanggil insert() semua nilai dari Entitas diteruskan ke metode, tetapi ketika kita memanggil update(), maka hanya nilai yang telah berubah yang diteruskan.

Mengisi Properti dengan Cepat

Class Entitas juga menyediakan metode, fill() yang memungkinkan Anda memasukkan array pasangan kunci/nilai ke dalam class dan mengisi properti class. Properti apa pun dalam array akan diatur pada Entitas. Namun, saat menyimpan melalui model, hanya field dalam $allowedFields yang benar-benar akan disimpan ke database, sehingga Anda dapat menyimpan data tambahan pada entitas Anda tanpa terlalu khawatir tentang field yang menyimpang yang disimpan secara tidak benar.

<?php

$data = $this->request->getPost();

$user = new \App\Entities\User();
$user->fill($data);
$userModel->save($user);

Anda juga dapat meneruskan data di konstruktor dan data akan diteruskan melalui metode fill() selama instansiasi.

<?php

$data = $this->request->getPost();

$user = new \App\Entities\User($data);
$userModel->save($user);

Mengakses Properti Secara Massal

Class Entitas memiliki dua metode untuk mengekstrak semua properti yang tersedia ke dalam array: toArray() dan toRawArray(). Menggunakan versi raw akan melewati metode "getter" ajaib dan casting. Kedua metode dapat mengambil parameter boolean pertama untuk menentukan apakah nilai yang dikembalikan harus difilter oleh nilai yang telah berubah, dan parameter boolean terakhir untuk membuat metode rekursif, jika ada Entitas bersarang.

Menangani Logika Bisnis

Meskipun contoh di atas nyaman, mereka tidak membantu menegakkan logika bisnis apa pun. Class Entitas dasar mengimplementasikan beberapa metode __get() dan __set() cerdas yang akan memeriksa metode khusus dan menggunakannya alih-alih menggunakan atribut secara langsung, memungkinkan Anda untuk menegakkan logika bisnis atau konversi data apa pun yang Anda butuhkan.

Berikut adalah entitas Pengguna yang diperbarui untuk memberikan beberapa contoh bagaimana ini dapat digunakan:

<?php

namespace App\Entities;

use CodeIgniter\Entity\Entity;
use CodeIgniter\I18n\Time;

class User extends Entity
{
    public function setPassword(string $pass)
    {
        $this->attributes['password'] = password_hash($pass, PASSWORD_BCRYPT);

        return $this;
    }

    public function setCreatedAt(string $dateString)
    {
        $this->attributes['created_at'] = new Time($dateString, 'UTC');

        return $this;
    }

    public function getCreatedAt(string $format = 'Y-m-d H:i:s')
    {
        // Convert to CodeIgniter\I18n\Time object
        $this->attributes['created_at'] = $this->mutateDate($this->attributes['created_at']);

        $timezone = $this->timezone ?? app_timezone();

        $this->attributes['created_at']->setTimezone($timezone);

        return $this->attributes['created_at']->format($format);
    }
}

Hal pertama yang perlu diperhatikan adalah nama metode yang telah kita tambahkan. Untuk setiap metode, class mengharapkan nama kolom snake_case dikonversi menjadi PascalCase, dan diawali dengan set atau get. Metode-metode ini kemudian akan secara otomatis dipanggil setiap kali Anda mengatur atau mengambil properti class menggunakan sintaks langsung (yaitu, $user->email). Metode-metode tersebut tidak perlu bersifat publik kecuali Anda ingin mereka diakses dari class lain. Misalnya, properti class created_at akan diakses melalui metode setCreatedAt() dan getCreatedAt().

Catatan

Ini hanya berfungsi saat mencoba mengakses properti dari luar class. Setiap metode internal ke class harus memanggil metode setX() dan getX() secara langsung.

Dalam metode setPassword() kita memastikan bahwa password selalu di-hash.

Dalam setCreatedAt() kita mengubah string yang kita terima dari model menjadi objek DateTime, memastikan bahwa zona waktu kita adalah UTC sehingga kita dapat dengan mudah mengkonversi zona waktu pengguna saat ini. Dalam getCreatedAt(), itu mengubah waktu menjadi string yang diformat dalam zona waktu aplikasi saat ini.

Meskipun cukup sederhana, contoh-contoh ini menunjukkan bahwa menggunakan class Entitas dapat memberikan cara yang sangat fleksibel untuk menegakkan logika bisnis dan membuat objek yang menyenangkan untuk digunakan.

<?php

// Auto-hash the password - both do the same thing
$user->password = 'my great password';
$user->setPassword('my great password');

Getter/Setter Khusus

Baru pada versi 4.4.0.

Misalnya, jika class induk Entitas Anda sudah memiliki metode getParent() yang didefinisikan, dan Entitas Anda juga memiliki kolom bernama parent, ketika Anda mencoba menambahkan logika bisnis ke metode getParent() di class Entitas Anda, metode tersebut sudah didefinisikan.

Dalam kasus seperti itu, Anda dapat menggunakan getter/setter khusus. Alih-alih getX()/setX(), atur _getX()/_setX().

Dalam contoh di atas, jika Entitas Anda memiliki metode _getParent(), metode tersebut akan digunakan ketika Anda mendapatkan $entity->parent, dan metode _setParent() akan digunakan ketika Anda mengatur $entity->parent.

Pemetaan Data

Di banyak titik dalam karier Anda, Anda akan menemukan situasi di mana penggunaan aplikasi telah berubah dan nama kolom asli di database tidak lagi masuk akal. Atau Anda menemukan bahwa gaya pengkodean Anda lebih menyukai properti class camelCase, tetapi skema database Anda membutuhkan nama snake_case. Situasi ini dapat dengan mudah ditangani dengan fitur pemetaan data class Entitas.

Sebagai contoh, bayangkan Anda memiliki Entitas Pengguna yang disederhanakan yang digunakan di seluruh aplikasi Anda:

<?php

namespace App\Entities;

use CodeIgniter\Entity\Entity;

class User extends Entity
{
    protected $attributes = [
        'id'         => null,
        'name'       => null, // Represents a username
        'email'      => null,
        'password'   => null,
        'created_at' => null,
        'updated_at' => null,
    ];
}

Bos Anda datang kepada Anda dan mengatakan bahwa tidak ada yang menggunakan nama pengguna lagi, jadi Anda beralih untuk hanya menggunakan email untuk login. Tetapi mereka ingin mempersonalisasi aplikasi sedikit, jadi mereka ingin Anda mengubah field name untuk mewakili nama lengkap pengguna sekarang, bukan nama pengguna mereka seperti yang terjadi saat ini. Untuk menjaga semuanya tetap rapi dan memastikan semuanya terus masuk akal di database, Anda membuat migrasi untuk mengganti nama field name menjadi full_name untuk kejelasan.

Mengabaikan betapa dibuat-buatnya contoh ini, kita sekarang memiliki dua pilihan tentang cara memperbaiki class Pengguna. Kita bisa memodifikasi properti class dari $name menjadi $full_name, tetapi itu akan memerlukan perubahan di seluruh aplikasi. Sebagai gantinya, kita dapat cukup memetakan kolom full_name di database ke properti $name, dan selesai dengan perubahan Entitas:

<?php

namespace App\Entities;

use CodeIgniter\Entity\Entity;

class User extends Entity
{
    protected $attributes = [
        'id'         => null,
        'full_name'  => null, // In the $attributes, the key is the db column name
        'email'      => null,
        'password'   => null,
        'created_at' => null,
        'updated_at' => null,
    ];

    protected $datamap = [
        // property_name => db_column_name
        'name' => 'full_name',
    ];
}

Dengan menambahkan nama database baru kita ke array $datamap, kita dapat memberi tahu class properti class mana yang harus diakses melalui kolom database. Kunci array adalah properti class untuk memetakannya, di mana nilai dalam array adalah nama kolom di database.

Dalam contoh ini, ketika model mengatur field full_name pada class Pengguna, itu sebenarnya menetapkan nilai tersebut ke properti $name class, sehingga dapat diatur dan diambil melalui $user->name. Nilai tersebut masih akan dapat diakses melalui $user->full_name asli juga, karena ini diperlukan agar model dapat mengambil data kembali dan menyimpannya ke database. Namun, unset() dan isset() hanya berfungsi pada properti yang dipetakan, $user->name, bukan pada nama kolom database, $user->full_name.

Catatan

Saat Anda menggunakan Pemetaan Data, Anda harus mendefinisikan metode set*() dan get*() untuk nama kolom database. Dalam contoh ini, Anda harus mendefinisikan setFullName() dan getFullName().

Mutator

Mutator Tanggal

Secara default, class Entitas akan mengubah field bernama created_at, updated_at, atau deleted_at menjadi instance Time setiap kali diatur atau diambil. Class Time menyediakan sejumlah besar metode yang membantu dengan cara yang tidak dapat diubah dan terlokalisasi.

Anda dapat menentukan properti mana yang secara otomatis dikonversi dengan menambahkan nama ke properti $dates:

<?php

namespace App\Entities;

use CodeIgniter\Entity\Entity;

class User extends Entity
{
    protected $dates = ['created_at', 'updated_at', 'deleted_at'];
}

Sekarang, ketika salah satu properti tersebut diatur, mereka akan dikonversi ke instance Time, menggunakan zona waktu aplikasi saat ini, seperti yang diatur di app/Config/App.php:

<?php

$user = new \App\Entities\User();

// Converted to Time instance
$user->created_at = 'April 15, 2017 10:30:00';

// Can now use any Time methods:
echo $user->created_at->humanize();
echo $user->created_at->setTimezone('Europe/London')->toDateString();

Casting Properti

Anda dapat menentukan bahwa properti di Entitas Anda harus dikonversi ke tipe data umum dengan properti $casts. Opsi ini harus berupa array di mana kunci adalah nama properti class, dan nilainya adalah tipe data yang harus di-cast.

Casting properti memengaruhi pembacaan (get) dan penulisan (set), tetapi beberapa tipe memengaruhi hanya pembacaan (get).

Casting Tipe Skalar

Properti dapat di-cast ke salah satu tipe data berikut: integer, float, double, string, boolean, object, array, datetime, timestamp, uri dan int-bool. Tambahkan tanda tanya di awal tipe untuk menandai properti sebagai nullable, yaitu, ?string, ?integer.

Catatan

int-bool dapat digunakan sejak v4.3.0.

Misalnya, jika Anda memiliki entitas Pengguna dengan properti is_banned, Anda dapat meng-cast-nya sebagai boolean:

<?php

namespace App\Entities;

use CodeIgniter\Entity\Entity;

class User extends Entity
{
    protected $casts = [
        'is_banned'          => 'boolean',
        'is_banned_nullable' => '?boolean',
    ];
}

Casting Array/Json

Casting Array/Json sangat berguna dengan field yang menyimpan array atau json yang diserialkan di dalamnya. Ketika di-cast sebagai:

  • sebuah array, mereka akan secara otomatis di-unserialize,

  • sebuah json, mereka akan secara otomatis diatur sebagai nilai dari json_decode($value, false),

  • sebuah json-array, mereka akan secara otomatis diatur sebagai nilai dari json_decode($value, true),

ketika Anda mengatur nilai properti. Tidak seperti tipe data lainnya yang dapat Anda cast properti, tipe cast:

  • array akan melakukan serialisasi,

  • json dan json-array akan menggunakan fungsi json_encode pada

nilai setiap kali properti diatur:

<?php

namespace App\Entities;

use CodeIgniter\Entity\Entity;

class User extends Entity
{
    protected $casts = [
        'options'        => 'array',
        'options_object' => 'json',
        'options_array'  => 'json-array',
    ];
}

<?php

$user    = $userModel->find(15);
$options = $user->options;

$options['foo'] = 'bar';

$user->options = $options;
$userModel->save($user);

Casting CSV

Jika Anda tahu Anda memiliki array datar dengan nilai-nilai sederhana, mengodekannya sebagai string serial atau JSON mungkin lebih kompleks daripada struktur aslinya. Casting sebagai Comma-Separated Values (CSV) adalah alternatif yang lebih sederhana akan menghasilkan string yang menggunakan lebih sedikit ruang dan lebih mudah dibaca oleh manusia:

<?php

namespace App\Entities;

use CodeIgniter\Entity\Entity;

class Widget extends Entity
{
    protected $casts = [
        'colors' => 'csv',
    ];
}

Disimpan di database sebagai "red,yellow,green":

<?php

$widget->colors = ['red', 'yellow', 'green'];

Catatan

Casting sebagai CSV menggunakan metode internal PHP implode dan explode dan mengasumsikan semua nilai aman untuk string dan bebas koma. Untuk cast data yang lebih kompleks coba array atau json.

Casting Kustom

Anda dapat mendefinisikan tipe konversi Anda sendiri untuk mendapatkan dan mengatur data.

Pertama, Anda perlu membuat class handler untuk tipe Anda. Misalnya, class akan terletak di direktori app/Entities/Cast:

<?php

namespace App\Entities\Cast;

use CodeIgniter\Entity\Cast\BaseCast;

// The class must inherit the CodeIgniter\Entity\Cast\BaseCast class
class CastBase64 extends BaseCast
{
    public static function get($value, array $params = [])
    {
        return base64_decode($value, true);
    }

    public static function set($value, array $params = [])
    {
        return base64_encode($value);
    }
}

Sekarang Anda perlu mendaftarkannya:

<?php

namespace App\Entities;

use CodeIgniter\Entity\Entity;

class MyEntity extends Entity
{
    // Specify the type for the field
    protected $casts = [
        'key' => 'base64',
    ];

    // Bind the type to the handler
    protected $castHandlers = [
        'base64' => Cast\CastBase64::class,
    ];
}

// ...

$entity->key = 'test'; // dGVzdA==
echo $entity->key;     // test

Jika Anda tidak perlu mengubah nilai saat mendapatkan atau mengatur nilai. Maka cukup jangan mengimplementasikan metode yang sesuai:

<?php

namespace App\Entities\Cast;

use CodeIgniter\Entity\Cast\BaseCast;

class CastBase64 extends BaseCast
{
    public static function get($value, array $params = [])
    {
        return base64_decode($value, true);
    }
}

Parameter

Dalam beberapa kasus, satu tipe tidak cukup. Dalam situasi ini, Anda dapat menggunakan parameter tambahan. Parameter tambahan ditunjukkan dalam tanda kurung siku dan dicantumkan dengan koma seperti type[param1, param2].

<?php

namespace App\Entities;

use CodeIgniter\Entity\Entity;

class MyEntity extends Entity
{
    // Define a type with parameters
    protected $casts = [
        'some_attribute' => 'class[App\SomeClass, param2, param3]',
    ];

    // Bind the type to the handler
    protected $castHandlers = [
        'class' => 'SomeHandler',
    ];
}

<?php

namespace App\Entities\Cast;

use CodeIgniter\Entity\Cast\BaseCast;

class SomeHandler extends BaseCast
{
    public static function get($value, array $params = [])
    {
        var_dump($params);
        /*
         * Output:
         * array(3) {
         *   [0]=>
         *   string(13) "App\SomeClass"
         *   [1]=>
         *   string(6) "param2"
         *   [2]=>
         *   string(6) "param3"
         * }
         */
    }
}

Catatan

Jika tipe casting ditandai sebagai nullable seperti ?bool dan nilai yang diteruskan tidak null, maka parameter dengan nilai nullable akan diteruskan ke handler tipe casting. Jika tipe casting memiliki parameter yang telah ditentukan, maka nullable akan ditambahkan ke akhir daftar.

Memeriksa Atribut yang Berubah

Anda dapat memeriksa apakah atribut Entitas telah berubah sejak dibuat. Satu-satunya parameter adalah nama atribut yang akan diperiksa:

<?php

$user = new \App\Entities\User();
$user->hasChanged('name'); // false

$user->name = 'Fred';
$user->hasChanged('name'); // true

Atau untuk memeriksa seluruh entitas untuk nilai yang berubah, hilangkan parameternya:

<?php

$user->hasChanged(); // true
