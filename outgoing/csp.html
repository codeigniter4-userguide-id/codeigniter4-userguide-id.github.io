Apa itu Content Security Policy?

Salah satu perlindungan terbaik yang Anda miliki terhadap serangan XSS adalah mengimplementasikan Content Security Policy (CSP) di situs. Ini mengharuskan Anda untuk menentukan dan mengotorisasi setiap sumber konten yang disertakan dalam HTML situs Anda, termasuk gambar, stylesheet, file JavaScript, dan sebagainya. Browser akan menolak konten dari sumber yang tidak disetujui secara eksplisit. Otorisasi ini didefinisikan dalam header Content-Security-Policy response dan menawarkan berbagai opsi konfigurasi.

Ini terdengar kompleks, dan di beberapa situs, tentu bisa menjadi tantangan. Namun, untuk banyak situs sederhana, di mana semua konten disajikan oleh domain yang sama (misalnya, http://example.com), sangat mudah untuk diintegrasikan.

Karena ini adalah subjek yang kompleks, panduan pengguna ini tidak akan membahas semua detailnya. Untuk informasi lebih lanjut, Anda harus mengunjungi situs-situs berikut:

  • Situs utama Content Security Policy

  • Spesifikasi W3C

  • Pengantar di HTML5Rocks

  • Artikel di SitePoint

Mengaktifkan CSP

Penting

Debug Toolbar dapat menggunakan Kint, yang mengeluarkan skrip inline. Oleh karena itu, ketika CSP diaktifkan, nonce CSP secara otomatis dikeluarkan untuk Debug Toolbar. Namun, jika Anda tidak menggunakan nonce CSP, ini akan mengubah header CSP menjadi sesuatu yang tidak Anda inginkan, dan akan berperilaku berbeda daripada di produksi; jika Anda ingin memverifikasi perilaku CSP, matikan Debug Toolbar.

Secara default, dukungan untuk ini dimatikan. Untuk mengaktifkan dukungan di aplikasi Anda, edit nilai CSPEnabled di app/Config/App.php:

<?php

namespace Config;

use CodeIgniter\Config\BaseConfig;

class App extends BaseConfig
{
    // ...

    public bool $CSPEnabled = true;
}

Ketika diaktifkan, objek response akan berisi instance CodeIgniter\HTTP\ContentSecurityPolicy. Nilai yang diatur di app/Config/ContentSecurityPolicy.php diterapkan ke instance tersebut, dan jika tidak ada perubahan yang diperlukan selama runtime, maka header yang diformat dengan benar akan dikirim dan Anda selesai.

Dengan CSP diaktifkan, dua baris header ditambahkan ke response HTTP: header Content-Security-Policy, dengan kebijakan yang mengidentifikasi jenis atau asal konten yang secara eksplisit diizinkan untuk konteks yang berbeda, dan header Content-Security-Policy-Report-Only, yang mengidentifikasi jenis atau asal konten yang akan diizinkan tetapi juga akan dilaporkan ke tujuan pilihan Anda.

Implementasi kami menyediakan perlakuan default, yang dapat diubah melalui metode reportOnly(). Ketika entri tambahan ditambahkan ke direktif CSP, seperti yang ditunjukkan di bawah, itu akan ditambahkan ke header CSP yang sesuai untuk pemblokiran atau pencegahan. Itu dapat ditimpa per panggilan, dengan memberikan parameter kedua opsional ke panggilan metode penambahan.

Konfigurasi Runtime

Jika aplikasi Anda perlu melakukan perubahan saat runtime, Anda dapat mengakses instance di $this->response->getCSP() di controller Anda.

Class ini memiliki sejumlah metode yang memetakan dengan cukup jelas ke nilai header yang sesuai yang perlu Anda atur. Contoh ditunjukkan di bawah, dengan kombinasi parameter yang berbeda, meskipun semuanya menerima direktif nama atau array dari mereka:

<?php

// get the CSP instance
$csp = $this->response->getCSP();

// specify the default directive treatment
$csp->reportOnly(false);

// specify the origin to use if none provided for a directive
$csp->setDefaultSrc('cdn.example.com');

// specify the URL that "report-only" reports get sent to
$csp->setReportURI('http://example.com/csp/reports');

// specify that HTTP requests be upgraded to HTTPS
$csp->upgradeInsecureRequests(true);

// add types or origins to CSP directives
// assuming that the default treatment is to block rather than just report
$csp->addBaseURI('example.com', true); // report only
$csp->addChildSrc('https://youtube.com'); // blocked
$csp->addConnectSrc('https://*.facebook.com', false); // blocked
$csp->addFontSrc('fonts.example.com');
$csp->addFormAction('self');
$csp->addFrameAncestor('none', true); // report this one
$csp->addImageSrc('cdn.example.com');
$csp->addMediaSrc('cdn.example.com');
$csp->addManifestSrc('cdn.example.com');
$csp->addObjectSrc('cdn.example.com', false); // reject from here
$csp->addPluginType('application/pdf', false); // reject this media type
$csp->addScriptSrc('scripts.example.com', true); // allow but report requests from here
$csp->addStyleSrc('css.example.com');
$csp->addSandbox(['allow-forms', 'allow-scripts']);

Parameter pertama untuk setiap metode "add" adalah nilai string yang sesuai, atau array dari mereka.

Hanya Laporan

Metode reportOnly() memungkinkan Anda untuk menentukan perlakuan pelaporan default untuk sumber-sumber berikutnya, kecuali jika ditimpa.

Misalnya, Anda dapat menentukan bahwa youtube.com diizinkan, dan kemudian menyediakan beberapa sumber yang diizinkan tetapi dilaporkan:

<?php

// get the CSP instance
$csp = $this->response->getCSP();

$csp->addChildSrc('https://youtube.com'); // allowed
$csp->reportOnly(true);
$csp->addChildSrc('https://metube.com'); // allowed but reported
$csp->addChildSrc('https://ourtube.com', false); // allowed

Hapus Direktif

Jika Anda ingin menghapus direktif CSP yang ada, Anda dapat menggunakan metode clearDirective():

<?php

// get the CSP instance
$csp = $this->response->getCSP();

$csp->clearDirective('style-src');

Konten Inline

Dimungkinkan untuk mengatur situs web agar tidak melindungi bahkan skrip dan gaya inline di halamannya sendiri, karena ini mungkin merupakan hasil dari konten yang dibuat pengguna. Untuk melindungi dari ini, CSP memungkinkan Anda untuk menentukan nonce dalam tag <style> dan <script>, dan untuk menambahkan nilai-nilai tersebut ke header response.

Menggunakan Placeholder

Ini adalah hal yang merepotkan untuk ditangani dalam kehidupan nyata, dan paling aman ketika dihasilkan secara on-the-fly. Untuk membuatnya sederhana, Anda dapat menyertakan placeholder {csp-style-nonce} atau {csp-script-nonce} dalam tag dan itu akan ditangani secara otomatis untuk Anda:

// Asli
<script {csp-script-nonce}>
    console.log("Skrip tidak akan berjalan karena tidak mengandung atribut nonce");
</script>

// Menjadi
<script nonce="Eskdikejidojdk978Ad8jf">
    console.log("Skrip tidak akan berjalan karena tidak mengandung atribut nonce");
</script>

// ATAU
<style {csp-style-nonce}>
    . . .
</style>

Peringatan

Jika penyerang menyuntikkan string seperti <script {csp-script-nonce}>, itu mungkin menjadi atribut nonce yang sebenarnya dengan fungsionalitas ini. Anda dapat menyesuaikan string placeholder dengan properti $scriptNonceTag dan $styleNonceTag di app/Config/ContentSecurityPolicy.php.

Menggunakan Fungsi

Jika Anda tidak menyukai fungsionalitas penggantian otomatis di atas, Anda dapat mematikannya dengan mengatur $autoNonce = false di app/Config/ContentSecurityPolicy.php.

Dalam kasus ini, Anda dapat menggunakan fungsi csp_script_nonce() dan csp_style_nonce():

// Asli
<script <?= csp_script_nonce() ?>>
    console.log("Skrip tidak akan berjalan karena tidak mengandung atribut nonce");
</script>

// Menjadi
<script nonce="Eskdikejidojdk978Ad8jf">
    console.log("Skrip tidak akan berjalan karena tidak mengandung atribut nonce");
</script>

// ATAU
<style <?= csp_style_nonce() ?>>
    . . .
</style>
